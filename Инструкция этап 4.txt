ЭТАП 4: ЗАПОЛНЕНИЕ TASKS

ГЛОССАРИЙ ТЕРМИНОВ

Термин | Описание
--------|----------
HOG (Hidden Object Game) | Скрытый предмет, который игрок должен найти в игровом мире
Task type | Тип задания, определяющий механику игрового процесса
Classname | Уникальный идентификатор квеста или предмета в системе
GR (Game Resource) | Игровой ресурс, который выпадает при определенных действиях
Collection | Коллекционные предметы, которые игрок собирает
is_silhouette | Параметр, указывающий, что предмет показан только силуэтом
is_hide | Параметр, скрывающий иконку предмета под знаком "?"
Mystery | Особый тип задания в форме загадки
in_guest | Параметр, указывающий, что действие выполняется в гостях у друзей

ПРАВИЛА НЕЙМИНГА:

Сущность | Формат | Пример
----------|--------|--------
Пакет квестов | [Тема]_[Год] | Easter_2025
Квесты | [Тема]_[Год]_Story_[Номер] | Easter_2025_Story_1
HOG | [Тема]_[Год]_HOG_[Номер] | Easter_2025_HOG_1
GR (ресурс) | [Тема]_[Год]_GR_[Номер] | Easter_2025_GR_1
GR из цветов | использовать общий счетчик GR | Easter_2025_GR_2
PER (запрос) | [Тема]_[Год]_PER_[Номер] | Easter_2025_PER_1
ASK (просьба) | [Тема]_[Год]_ASK_[Номер] | Easter_2025_ASK_1
Рецепты (R) | [Тема]_[Год]_R_[Номер] | Easter_2025_R_1
Персонажи | [Тема]_[Год]_Character_[Номер] | Easter_2025_Character_1
Свободные подарки | [Тема]_[Год]_FG_[Номер] | Easter_2025_FG_1

Требования к этапу:
- Заполнить JSON-структуру для каждого таска
- Следовать шаблону для выбранного Task type
- Адаптировать содержимое под контекст задания
- Соблюдать правила нейминга
- Для загадок (mystery) создавать рифмованные тексты

Ограничения:
1. Нельзя менять текст который не находится в [] в шаблонах это стандартный текст и должен остаться как есть.

НОВЫЙ WORKFLOW ЭТАПА 4

Этап 4 теперь работает не напрямую с raw-данными и не с полным индексом.

Входные файлы:
- `output/context_pack.json` — основной вход для ИИ. В нем есть квесты, таски, выбранные `task_template_id`, контекст сюжета и компактные списки реальных кандидатов.
- `data/task_templates.json` — машинный каталог актуальных шаблонов `TT-001` ... `TT-035`.
- `output/filled_tasks.example.json` — пример формата ответа.

Результат этапа 4:
- `output/filled_tasks.json`

После заполнения обязательно запустить проверку:

```bash
python src/validate_task_objects.py output/filled_tasks.json
```

Если валидатор нашел ошибки, нужно исправить `output/filled_tasks.json` или попросить ИИ перезаполнить конкретные task.

ГЛАВНОЕ ПРАВИЛО ЭТАПА 4

ИИ может творчески адаптировать `title`, `hint`, загадку и смысловое обоснование, но не может выдумывать реальные игровые данные.

Нельзя выдумывать:
- `garbage_classname`
- `garbage_title`
- `flower_classname`
- `flower_title`
- `collection_classname`
- `collection_title`
- реальные location title

Все эти значения должны быть взяты только из `candidates` внутри `output/context_pack.json`.

Можно создавать по неймингу пакета:
- `GR`
- `ASK`
- `PER`
- `CL`
- `FA`
- `HOG`
- `R`
- `Character`
- `Stuff`
- action classname для dialog/give/craft/post_photo

СТРОГИЕ ПРАВИЛА РЕАЛЬНЫХ ЛОКАЦИЙ

Любая фраза в `hint`, которая указывает место, должна ссылаться только на существующую игровую локацию.

Строго проверяемые фразы:
- `Он находится ...`
- `Она находится ...`
- `Место поиска: ...`
- `Найди его у себя на ...`

Правило:
- После этих фраз нельзя писать сюжетные или атмосферные места.
- Нельзя писать: `у крыльца`, `под тучей`, `в мясном дожде`, `у котла`, `на небесной трещине`, если такой локации нет в `context_pack`.
- Нужно использовать точный `title` реальной локации из `candidate.locations`.

Примеры:

Плохо:
```json
"hint": "Поговори с Дедушкой Домоведом. Для этого просто кликни на него. Он находится у крыльца под фрикадельковой тучей."
```

Хорошо:
```json
"hint": "Поговори с Дедушкой Домоведом. Для этого просто кликни на него. Он находится в Барной комнате."
```

Плохо:
```json
"hint": "Найди Следы мясного дождя. Место поиска: крыльцо у дома. Если найти все не удаётся, можно купить подсказку."
```

Хорошо:
```json
"hint": "Найди Следы мясного дождя. Место поиска: Барная комната. Если найти все не удаётся, можно купить подсказку."
```

Если для generated/story template нет своего candidate, используй реальную локацию из других кандидатов этого же квеста в `context_pack`.
Если подходящей реальной локации нет, не выдумывай ее: нужно запросить новый context pack или ручной override.

ПРАВИЛО РАЗДЕЛЕНИЯ ИСТОЧНИКОВ

В одном квесте не нужно ставить:
- task на уборку конкретного мусора;
- и task на получение элемента коллекции, который выпадает из этого же мусора.

Плохо:
```text
TT-021 garbage FoxTrail
TT-011 collection_drop FoxTrailCollection5 из FoxTrail
```

Лучше:
```text
TT-021 garbage FoxTrail
TT-011 collection_drop из другого source_classname
```

Причина: игроку это выглядит как повтор одного и того же источника, а квест должен давать разнообразные действия.

ПРАВИЛО НУМЕРАЦИИ GENERATED-СУЩНОСТЕЙ

Нумерация generated-сущностей идет отдельными списками по типу сущности, а не по номеру task.

Для пакета `MeatballRain_2026` правильно:

```text
MeatballRain_2026_HOG_1
MeatballRain_2026_GR_1
MeatballRain_2026_ASK_1
MeatballRain_2026_GR_2
MeatballRain_2026_R_1
```

Неправильно:

```text
MeatballRain_2026_HOG_2
MeatballRain_2026_GR_6
MeatballRain_2026_ASK_7
MeatballRain_2026_GR_8
MeatballRain_2026_R_9
```

Отдельные счетчики:
- HOG: `[Префикс]_HOG_1`, `[Префикс]_HOG_2`, ...
- GR: `[Префикс]_GR_1`, `[Префикс]_GR_2`, ...
- ASK: `[Префикс]_ASK_1`, `[Префикс]_ASK_2`, ...
- PER: `[Префикс]_PER_1`, `[Префикс]_PER_2`, ...
- CL: `[Префикс]_CL_1`, `[Префикс]_CL_2`, ...
- FA: `[Префикс]_FA_1`, `[Префикс]_FA_2`, ...
- R: `[Префикс]_R_1`, `[Префикс]_R_2`, ...

Для `GR с цветов` тоже используй общий счетчик `GR`, если отдельная инструкция пакета не требует иначе.

ПРАВИЛО ВИЗУАЛИЗИРУЕМЫХ ПРЕДМЕТОВ

Generated-предметы `GR`, `ASK`, `PER`, `CL`, `FA`, `R` должны быть конкретными объектами, которые можно нарисовать и добавить в игру.

Нельзя делать предметом абстрактную сущность:
- `Небесное давление`
- `Магическая сила`
- `Сытость`
- `Заряд радости`
- `Фрикадельковый фронт`

Нужно делать предметом физический объект:
- `Латунная стрелка барометра`
- `Крышка для непослушного котла`
- `Лепесток сытости`
- `Соусный клапан`
- `Медный фиксатор трещины`

Если хочется передать абстрактную идею, привяжи ее к предмету:
- не `Небесное давление`, а `Латунная стрелка барометра`;
- не `Сытость`, а `Лепесток сытости`;
- не `Магия котла`, а `Закопченный амулет котла`.

ФОРМАТ `filled_tasks.json`

Ответ этапа 4 должен быть JSON-объектом:

```json
{
  "quests": [
    {
      "classname_quests": "[classname квеста]",
      "title_quest": "[название квеста]",
      "quest_number": 1,
      "tasks": [
        {
          "task_number": 1,
          "task_template_id": "TT-020",
          "task_template_name": "Уборка конкретного мусора в гостях",
          "task_type": "garbage classname in_guest",
          "selected_candidate_id": "garbage:Ashes",
          "choice_reason": "Почему выбран именно этот кандидат по контексту квеста.",
          "task_object": {
            "type": "garbage",
            "classname": "Ashes",
            "in_guest": 1,
            "amount": 1,
            "price": 1,
            "title": "Убери мусор Пепел в гостях",
            "hint": "Убери мусор Пепел в гостях. Для этого просто кликни на нужный мусор в гостях у друга. Место поиска: Котельная.",
            "identifier": ""
          }
        }
      ]
    }
  ]
}
```

Обязательные поля task-записи:
- `task_number`
- `task_template_id`
- `task_template_name`
- `task_type`
- `selected_candidate_id`
- `choice_reason`
- `task_object`

Если шаблон не требует реального кандидата, например `TT-001`, `TT-002`, `TT-008`, `TT-009`, `TT-033`, `TT-034`, тогда:
- `selected_candidate_id` можно поставить `null`;
- игровые/generated classname создаются по правилам нейминга;
- в `choice_reason` нужно объяснить сюжетную логику.

КАК ВЫБИРАТЬ КАНДИДАТА

Для каждого task в `context_pack.json` есть:
- `task_template_id`
- `task_template_name`
- `task_type`
- `description`
- `congratulation`
- `character`
- `candidate_domain`
- `candidates`
- `manual_override`
- `notes`

Правила выбора:
1. Выбери ровно один `selected_candidate_id` из `candidates`, если список `candidates` не пустой.
2. Не выбирай кандидата вне списка.
3. Если есть `manual_override.avoid_candidates`, не выбирай эти `candidate_id`.
4. Если есть `manual_override.prefer_candidates`, сначала попробуй выбрать из них, но только если это подходит контексту.
5. Если есть `manual_override.instruction`, выполни эту инструкцию.
6. Подбирай кандидата по смыслу `description`, `congratulation`, `title_quest` и персонажа.
7. В `choice_reason` коротко объясни выбор.

Как использовать кандидатов:

Для `candidate_domain = "garbage"`:
- `task_object.classname` должен быть равен `candidate.garbage_classname`;
- в title/hint используй `candidate.garbage_title`;
- место поиска бери из `candidate.locations`.

Для `candidate_domain = "flower"`:
- `task_object.param` должен быть равен `candidate.flower_classname`;
- в title/hint используй `candidate.flower_title`.

Для `candidate_domain = "collection_drop"`:
- `task_object.classname` должен быть равен `candidate.collection_classname`;
- если есть `icon`, он должен быть равен `candidate.collection_classname`;
- в title/hint используй `candidate.collection_title`;
- источник выпадения бери из `candidate.source_title`, `candidate.source_type`, `candidate.mode`, `candidate.locations`.

Для `candidate_domain = "gr_garbage"`:
- `task_object.classname` и `icon` создаются как `[Префикс]_GR_[Номер]`;
- в hint используй `candidate.garbage_title`;
- место поиска бери из `candidate.locations`;
- сам GR — новый ресурс пакета, но источник мусора должен быть реальным кандидатом.

Для generated/story templates без кандидатов:
- используй контекст квеста;
- соблюдай нейминг;
- не вставляй реальные garbage/flower/collection, если их нет в кандидате.

ТИПИЧНЫЙ ПОРЯДОК РАБОТЫ ИИ НА ЭТАПЕ 4

1. Открыть `output/context_pack.json`.
2. Для каждого квеста прочитать `title_quest`, `description`, `congratulation`, `Character`.
3. Для каждого task посмотреть `task_template_id`.
4. Найти соответствующий шаблон в этой инструкции.
5. Выбрать кандидата из `candidates`, если он нужен.
6. Заполнить `task_object`.
7. Проверить, что реальные classname/title взяты только из выбранного кандидата.
8. Сохранить итоговый JSON в `output/filled_tasks.json`.
9. Запустить `python src/validate_task_objects.py output/filled_tasks.json`.

КРИТИЧЕСКИЕ ОШИБКИ ЭТАПА 4

Нельзя:
- выбирать `selected_candidate_id`, которого нет в `context_pack`;
- менять `task_template_id` на этапе 4;
- использовать `TT-035`;
- придумывать реальные collection/garbage/flower classname;
- придумывать реальные локации в `hint`;
- писать после `Он находится` или `Место поиска:` локацию, которой нет в `context_pack`;
- использовать collection title от одного кандидата, а classname от другого;
- использовать garbage title от одного кандидата, а classname от другого;
- ставить в одном квесте уборку мусора и collection drop из того же самого мусора;
- для flower task ставить `param`, которого нет в выбранном flower candidate;
- нумеровать generated-сущности по номеру task вместо отдельного счетчика сущности;
- делать `GR`, `ASK`, `PER`, `R` абстрактными сущностями, которые нельзя отрисовать как предмет;
- игнорировать `manual_override`;
- возвращать CSV вместо `filled_tasks.json`;
- удалять обязательные поля шаблона.

ШАБЛОНЫ ДЛЯ РАЗНЫХ ТИПОВ ТАСКОВ:

Источник актуального списка: таблица `Шаблоны тасков.csv`.

ID шаблона таска — стабильный идентификатор шаблона для этапа 4. Русское название шаблона сохранено как в таблице.

Важно:
- `task_template_id` не является полем игрового task object. Это служебный ID инструкции.
- Все значения в квадратных скобках нужно заполнить по контексту квеста и реальным данным.
- Мусор, цветы, коллекции, локации и их title брать только из `data/quest_ready_index.json` и `data/quest_ready_drops.index.json`.
- Пакетные сущности квеста (`HOG`, `GR`, `ASK`, `PER`, `CL`, `FA`, `R`, `Character`, `Stuff`) создаются по правилам нейминга текущего квестового пакета.
- Поле `identifier` присутствует в актуальной таблице как пустое служебное поле. Если pipeline не требует его в JSON task object, оно может оставаться пустым.
- `amount` и `price` в таблице отличаются по примерам; если отдельная балансировочная таблица не задана, подбирай их осознанно под сложность задания.
